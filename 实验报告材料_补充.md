## 一、Bug 修复

### 1.1 MySQL 密码环境适配问题

**现象**：项目从其他电脑迁移后无法启动，`start.bat` 运行后数据库连接失败。

**根因**：`application.properties` 和 JAR 包中数据库密码硬编码为 `root`，目标电脑 MySQL root 密码为 `123456`。

**修复**：
- 将 MySQL root 密码改回 `root`：`ALTER USER 'root'@'localhost' IDENTIFIED BY 'root'`
- 执行 `init.sql` 创建 `cogniflow` 数据库及12张表
- `start.bat` 中 mysqladmin ping 密码同步为 `root`

### 1.2 AuthController 登录后 Session 未写入（核心 Bug）

**现象**：能登录成功，但点击任何页面（课程、考试、计划等）就跳回登录页。

**根因**：`AuthController.login()` 验证密码后只返回了 JSON 用户信息，**漏写了 `request.getSession().setAttribute("loginUser", user)`**。导致 `AuthInterceptor` 每次拦截检查时 Session 中 `loginUser` 为 null，判定未登录重定向到 `/login`。

**修复**（`AuthController.java:20-27`）：
```java
@PostMapping("/login")
public Map<String, Object> login(@RequestBody Map<String, String> body, HttpServletRequest request) {
    User user = userService.login(body.get("username"), body.get("password"));
    request.getSession().setAttribute("loginUser", user);  // ← 新增
    return Map.of("code", 200, ...);
}
```

### 1.3 计划保存按钮偶尔不出现

**现象**：AI 生成计划后保存按钮"💡 点击这里保存到「我的计划」"不显示。

**根因**：`index.html` 中 `renderRoadmap()` 仅当 AI 回复包含 ≥5 个 "Day N:" 行才显示保存按钮。

**修复**：改为 AI 每次回复后都显示保存按钮，不再限制格式：
```javascript
lastPlanContent = data.reply;
const days = parsePlanDays(data.reply);
if (days.length >= 5) renderRoadmap(data.reply);
else document.getElementById('saveHint').style.display = 'block';  // ← 保证始终可保存
```

### 1.4 计划编辑后进度不刷新（24/23 = 104%）

**现象**：编辑计划内容后线路图 Day 条数变少，但 `plans` 列表页进度没变，详情页出现 24/23 = 104%。

**根因**：
1. `plans.html` 进度分母用的是数据库存的 `totalDays`，编辑后未更新
2. `progressDays` 未钳制到 totalDays 以内

**修复**：
- 后端 `updateContent()` 同时更新 `totalDays`，并钳制 `progressDays ≤ totalDays`
- 前端 `saveEdit()` 发送 `totalDays: newDays.length`
- `plan-detail.html` 进度分母改为 `planDays.length`（实际解析数）

### 1.5 parsePlanDays 正则不兼容 AI 格式

**现象**：线路图解析失败，AI 用粗体 `**Day 1**:xxx` 或列表 `- Day 1: xxx` 格式时匹配不到。

**修复**（`plan-detail.html` & `index.html`）：
```javascript
function parsePlanDays(content) {
    const days = [];
    for (let line of lines) {
        let clean = line.replace(/^\*\*|\*\*$/g, '').replace(/^[\s\-•*]+/, '').trim();
        let m = clean.match(/^Day\s*(\d+)[：:]\s*(.+)/);
        if (m) {
            let label = m[2].trim().replace(/\*+$/g, '').trim();
            if (label) days.push({ day: parseInt(m[1]), label: label.substring(0, 60) });
        }
    }
    return days;
}
```
- 去掉行首粗体标记 `**` 和列表符 `- • *`
- 过滤空标签（`Day 23: ` 后面没内容）
- 排除异常格式（`Day 24及之后`）

## 二、数据库设计优化

### 2.1 删除课程不再 CASCADE 级联丢失用户数据

**原设计问题**：`courses → knowledge_points` 和 `courses → exam_records` 均为 CASCADE。删课程后学生的知识点学习记录和考试记录全部消失。

**修改**：
```sql
-- knowledge_points: CASCADE → SET NULL
ALTER TABLE knowledge_points DROP FOREIGN KEY fk_kp_course;
ALTER TABLE knowledge_points ADD CONSTRAINT fk_kp_course
    FOREIGN KEY (course_id) REFERENCES courses(id) ON DELETE SET NULL;

-- exam_records: CASCADE → SET NULL  
ALTER TABLE exam_records DROP FOREIGN KEY fk_er_course;
ALTER TABLE exam_records ADD CONSTRAINT fk_er_course
    FOREIGN KEY (course_id) REFERENCES courses(id) ON DELETE SET NULL;
```

**效果**：删除课程后，知识点和考试记录保留，课程名显示为 `(已删除课程)`。

### 2.2 题库表删除知识点外键

**原因**：`questions.knowledge_point_id` 实际使用中始终为 NULL（AI 生成题库时未返回知识点映射）。

**修改**：
```sql
ALTER TABLE questions DROP FOREIGN KEY fk_q_kp;
ALTER TABLE questions DROP COLUMN knowledge_point_id;
```
同步删除 `Question.java` 实体中的 `knowledgePointId` 字段及 getter/setter，清理前端题库弹窗中的知识点下拉选择器。

### 2.3 更新后完整外链关系

```
courses → knowledge_points   SET NULL  (原CASCADE)
courses → exam_records       SET NULL  (原CASCADE)
courses → questions          CASCADE   (不变)
users → study_records        CASCADE   (不变)
users → exam_records         CASCADE   (不变)
users → announcements        SET NULL  (不变)
users → system_logs          SET NULL  (不变)
knowledge_points → study_records  CASCADE (不变)
exam_records → exam_answers  CASCADE   (不变)
```

## 四、技术总结补充

### 4.1 关于 `exam_records.question_ids` TEXT 存储逗号分隔 ID

这是项目中**唯一的反范式设计**（违反 1NF）。采用的理由是**考试快照完整性**：
- 考试记录需要保留考试那一刻的题目列表
- 题目可能在之后被修改或删除
- 用 TEXT 存储当时抽到的题目 ID，配合 `exam_answers` 表存答题明细，实现完整追溯

大厂做法会用 JSON 字段或 `exam_question_snapshots` 快照表，本项目的方案适合课设规模，核心权衡合理。

### 4.2 数据库范式分析

所有 12 张表满足 3NF，`exam_records.question_ids` 为有意的反范式设计（见上）。删除了无用的 `questions.knowledge_point_id` 后，整体设计更干净。

### 4.3 Maven 的作用

Maven 只在修改源代码后需要重新打包时使用。JAR 包内嵌了 Tomcat、Spring Boot、所有依赖库（含 MySQL 驱动）、HTML 模板和配置文件。运行时只需 JDK 和 MySQL，不需要额外安装 Spring Boot、Tomcat、Maven 等。

**打包命令**：`mvn clean package -DskipTests`
**本地 Maven 路径**：`C:\tools\apache-maven-3.9.16`
